## 第二部分 2.3 软件系统介绍

### 2.3.1 软件整体介绍

本系统软件基于 ESP-IDF v5.5.4 实时操作系统开发，采用 FreeRTOS 多任务架构，整体分为 **6 个功能子系统**，通过信号量、任务通知和事件队列实现模块间松耦合通信。

**软件架构总览：**

```
┌─────────────────────────────────────────────────────────┐
│                    应用层 (Application)                    │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │ 售货机 UI │ │ 人脸管理 │ │ 语音控制 │ │ Web 服务 │    │
│  │ (LVGL)   │ │ (SD卡)   │ │ (ESP-SR) │ │ (HTTP)   │    │
│  └────┬─────┘ └────┬─────┘ └────┬─────┘ └────┬─────┘    │
│       │            │            │            │           │
│  ┌────┴────────────┴────────────┴────────────┴─────┐     │
│  │              主控调度 (main.cpp)                  │     │
│  │         状态机 + 信号量 + 任务通知                │     │
│  └────┬────────────┬────────────┬────────────┬─────┘     │
│       │            │            │            │           │
│  ┌────┴─────┐ ┌────┴─────┐ ┌────┴─────┐ ┌────┴─────┐    │
│  │ 摄像头   │ │ 人脸检测 │ │ SPI LCD  │ │ WiFi     │    │
│  │ MIPI-CSI │ │ WHO框架  │ │ +触摸    │ │ ESP-Host │    │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘    │
│                    驱动层 (Driver)                        │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐ ┌──────────┐    │
│  │ OV5647   │ │ ESP-DL   │ │ ST7789   │ │ SDMMC    │    │
│  │ ISP/CSI  │ │ 模型推理 │ │ XPT2046  │ │ SD卡     │    │
│  └──────────┘ └──────────┘ └──────────┘ └──────────┘    │
│                    硬件层 (Hardware)                      │
└─────────────────────────────────────────────────────────┘
```

**核心调度流程：**

系统采用事件驱动的状态机模型，主循环逻辑如下：

1. **启动阶段**：检查唤醒原因（冷启动/触摸唤醒/定时器唤醒）→ 挂载 SD 卡 → 加载人脸数据库
2. **感知阶段**：启动 MIPI-CSI 摄像头管线 → 运行人脸检测（5 FPS）→ 等待检测信号
3. **交互阶段**：检测到人脸 → 暂停摄像头 → 初始化 LCD + 语音 → 运行售货机 UI
4. **执行阶段**：用户选择商品 → 支付确认 → 舵机出货 → 记录日志到 SD 卡
5. **恢复阶段**：停止语音/LCD → 恢复摄像头管线 → 回到感知阶段
6. **休眠阶段**：10 秒无人脸 → 释放摄像头/ISP 硬件 → 进入深度睡眠

**任务调度策略：**

系统共运行 7 个 FreeRTOS 任务，采用优先级抢占式调度：

| 任务名 | 优先级 | 核心 | 功能 |
|--------|--------|------|------|
| 帧采集 (WhoFrameCap) | 6 | 任意 | 从 OV5647 读取帧数据到 PSRAM |
| 人脸检测 (WhoDetect) | 5 | 任意 | MSRMNP_S8_V1 模型推理 |
| 语音采集 (voice_feed) | 5 | Core 0 | 从 I2S 麦克风读取音频 |
| 语音识别 (voice_detect) | 5 | Core 1 | MultiNet7 命令词识别 |
| 售货机 UI | 4 | 任意 | LVGL 界面交互 |
| 人脸监控 | 3 | 任意 | 空闲检测与深度睡眠控制 |
| LVGL 渲染 | 可配置 | 任意 | 驱动 lv_timer_handler() |

### 2.3.2 软件各模块介绍

#### 模块一：摄像头帧采集管线

**功能说明：** 基于 WHO (WebRTC Hands-free Online) 框架的帧采集节点图，负责从 OV5647 传感器获取图像帧并存入 PSRAM 缓冲区。

**关键输入/输出：**
- 输入：OV5647 MIPI-CSI 原始 Bayer 帧
- 输出：PSRAM 中的帧缓冲区指针 (`cam_fb`)

**流程：**
1. `WhoFrameCap` 创建节点图，添加 `WhoFetchNode` 节点
2. `WhoOV5647Cam` 初始化 MIPI-CSI 接口、ISP 图像处理器、传感器寄存器
3. 帧采集任务以优先级 6 运行，持续将传感器数据写入 PSRAM
4. 提供 `cam_fb_peek()` 接口供下游模块无锁读取最新帧

**暂停/恢复机制：** `pause()` 暂停帧采集释放 CPU，`resume()` 恢复采集。售货机 UI 运行期间暂停，完成后自动恢复。

#### 模块二：人脸检测与识别

**功能说明：** 使用 ESP-DL 框架运行轻量级人脸检测模型，检测到人脸后同步执行特征提取与匹配。

**关键输入/输出：**
- 输入：帧缓冲区图像 (`cam_fb`)
- 输出：检测结果（人脸框坐标 + 置信度）、识别结果（user_id + 相似度）

**流程：**
1. `WhoDetect` 任务以 5 FPS 速率从帧管线获取最新帧
2. 运行 `MSRMNP_S8_V1` 模型进行人脸检测
3. 检测回调 `on_face_detected()` 中同步调用 `HumanFaceRecognizer::recognize()`
4. 若匹配已知人脸 → `face_id_find_or_create()` 查找 user_id
5. 若未知人脸 → `face_id_register()` 注册新用户 + `enroll()` 入库特征
6. 释放信号量通知 `face_monitor_task`

**模型参数：**
- 检测模型：MSRMNP_S8_V1（MSR 多尺度残差网络，INT8 量化）
- 识别模型：MFN S8 V1（MobileFaceNet，INT8 量化）
- 特征维度：128 维浮点向量
- 相似度阈值：0.7（余弦相似度）

#### 模块三：人脸 ID 管理器

**功能说明：** 管理 SD 卡上的人脸数据库和购买日志，提供用户注册、查找、日志记录功能。

**关键输入/输出：**
- 输入：识别结果 (feat_id, similarity)
- 输出：user_id、购买记录 CSV

**数据格式：**

人脸数据库 (`/sdcard/face_db.csv`)：
```
id,name,enroll_time_us
1,User_1,1234567890
2,User_2,1234567900
```

购买日志 (`/sdcard/purchase_log.csv`)：
```
timestamp_us,face_id,product_name,price_text
1234568000,1,Cola,$2.00
```

**流程：**
1. `face_id_manager_init()` 挂载 SD 卡（SDMMC Slot 0, 4-bit），加载已有数据库
2. `face_id_register()` 分配递增 ID，写入 CSV
3. `face_id_find_or_create()` 查找已有 ID 或创建新用户
4. `face_id_log_purchase()` 追加购买记录到日志文件

#### 模块四：售货机 UI

**功能说明：** 基于 LVGL v9 的图形界面，实现完整的售货交互流程。

**界面状态机：**

```
┌──────────────┐    点击商品    ┌──────────────┐
│  商品选择     │──────────────→│  订单确认     │
│  (2×2 卡片)  │               │  (大图标)    │
└──────────────┘               └──────┬───────┘
       ↑                              │ 确认
       │ 取消/超时                     ↓
       │                        ┌──────────────┐
       │                        │  支付处理     │
       │                        │  (2s 动画)   │
       │                        └──────┬───────┘
       │                               │ 自动跳转
       │                               ↓
       │                        ┌──────────────┐
       │                        │  出货中       │
       │                        │  (舵机驱动)  │
       │                        └──────┬───────┘
       │                               │ 完成
       │                               ↓
       │                        ┌──────────────┐
       └────────────────────────│  感谢界面     │
                                │  (2s 自动返回)│
                                └──────────────┘
```

**关键设计：**
- 商品卡片采用 Flex 布局，2×2 网格排列
- 每个卡片包含彩色图标面板、商品名称、价格标签
- 支付界面使用 LVGL Spinner 动画模拟处理过程
- 出货时驱动对应 GPIO 的舵机转至 90°，1.5 秒后返回 0°
- 所有界面切换使用 `load_screen()` 安全删除旧界面，避免内存泄漏

#### 模块五：语音控制

**功能说明：** 基于 ESP-SR 的离线中文语音命令识别，支持 4 类商品的语音选购。

**关键输入/输出：**
- 输入：I2S 麦克风音频流 (16kHz, 16-bit)
- 输出：命令 ID + 商品名称

**流程：**
1. `voice_control_init()` 加载 SR 模型、初始化音频硬件（ES8311 + ES7210）、创建 AFE
2. `voice_control_start()` 创建两个任务：
   - `voice_feed_task`（Core 0）：从 I2S 读取音频送入 AFE
   - `voice_detect_task`（Core 1）：从 AFE 获取处理后音频，运行 MultiNet7 检测
3. 检测到命令词后调用回调函数，映射到商品索引
4. `voice_control_stop()` 设置退出标志，等待任务自行终止，销毁 AFE

**命令词映射：**

| 命令词 | 商品 | 索引 |
|--------|------|------|
| 可乐/来瓶可乐/我要可乐 | Cola | 0 |
| 雪碧/来瓶雪碧/我要雪碧 | Sprite→Water | 1 |
| 水/矿泉水/来瓶水 | Water | 1 |
| 薯片/来包薯片/我要薯片 | Chips | 2 |

**I2C 总线共享：** 音频编解码器（ES8311/ES7210）与摄像头（OV5647）共享 GPIO7/8 的 I2C 总线，通过 `camera_get_i2c_bus_handle()` 获取总线句柄实现复用。

#### 模块六：SPI LCD + 触摸驱动

**功能说明：** 管理 ST7789 LCD 显示和 XPT2046 触摸输入，集成 LVGL 图形库。

**关键设计：**
- SPI 总线初始化时设置 GPIO 驱动能力为 40mA（`GPIO_DRIVE_CAP_3`），确保高速通信稳定
- LVGL 使用双缓冲 DMA 渲染，缓冲区通过 `spi_bus_dma_memory_alloc()` 分配
- DMA 传输完成通过信号量同步（ISR 中 `xSemaphoreGiveFromISR`）
- 触摸输入包含去抖逻辑：按下需 2 次连续确认，释放需 3 次连续确认
- SPI 读取频率限制为 ≥5ms 间隔，避免阻塞显示刷新

**深度睡眠支持：** `spi_lcd_touch_deinit_display_only()` 仅释放 LCD 面板和 LVGL 资源，保留 XPT2046 触摸控制器和 SPI 总线，支持触摸唤醒。

#### 模块七：Web 服务器

**功能说明：** 提供 HTTP 状态面板和 MJPEG 摄像头实时流，支持远程监控。

**API 端点：**

| 端点 | 方法 | 功能 |
|------|------|------|
| `/` | GET | HTML 状态仪表板 |
| `/status` | GET | JSON 系统状态（运行时间、内存、人脸数等） |
| `/stream` | GET | MJPEG 多部分实时流（~8 FPS） |
| `/snapshot` | GET | 单帧 JPEG 快照 |

**JPEG 编码：** 使用 ESP32-P4 硬件 JPEG 编码器，将 RGB565 帧编码为 JPEG，质量参数 75，不占用 CPU 算力。

**WiFi 连接：** 通过 ESP-Hosted 组件连接 ESP32-C6 协处理器（SDIO 接口），实现 WiFi STA 模式。最大重试 5 次，连接超时 15 秒。

#### 模块八：深度睡眠与唤醒

**功能说明：** 管理系统低功耗状态，实现触摸唤醒和定时器保底唤醒。

**进入睡眠流程：**
1. 停止 Web 服务器
2. 停止摄像头管线，释放 ISP 硬件（`camera_deinit()`）
3. 配置 RTC 域保持供电（内部上拉需要）
4. 关闭 LCD 背光
5. 仅释放显示资源，保留触摸控制器
6. 配置 GPIO 2 为输入+上拉，等待触摸释放（最长 5 秒）
7. 配置 EXT1 唤醒源（GPIO 2 低电平）+ 定时器唤醒（10 秒）
8. 调用 `esp_deep_sleep_start()`

**唤醒处理：**
- EXT1 唤醒：等待 500ms 防抖，防止立即重新进入睡眠
- 定时器唤醒：正常启动流程
- 冷启动：正常启动流程

## 第四部分 心得体会

本作品从立项到完成历时约两个月，期间经历了硬件选型、驱动调试、算法移植、界面开发和系统联调等多个阶段，收获颇丰。

**硬件调试方面**，ESP32-P4 作为一款较新的芯片，社区资源相对有限，许多外设驱动需要参考官方例程和数据手册自行适配。MIPI-CSI 摄像头接口的调试是最大的挑战之一——从传感器寄存器配置到 ISP 图像管线的建立，每一步都需要仔细对照时序图和寄存器手册。I2C 总线的共享（摄像头与音频编解码器共用 GPIO7/8）也带来了设备地址冲突的风险，最终通过合理的总线仲裁机制解决了问题。

**算法移植方面**，人脸检测和识别模型的 INT8 量化版本在 ESP-DL 框架下运行良好，但需要注意模型输入尺寸与摄像头分辨率的匹配。初始版本使用 640×480 分辨率，检测效果不理想；改为 800×800 后，检测率显著提升，但帧缓冲区占用也随之增加到 ~2.4MB，必须使用 PSRAM。人脸识别的相似度阈值经过反复测试，最终设定为 0.7，在误识率和拒识率之间取得了较好的平衡。

**软件架构方面**，采用 FreeRTOS 多任务架构使得各模块可以独立开发和测试。信号量和任务通知机制保证了模块间的同步和通信效率。最值得总结的经验是"暂停/恢复"机制的设计——售货机 UI 运行时暂停摄像头管线，不仅释放了 CPU 算力给 LVGL 渲染，还避免了 MIPI-CSI 和 SPI 总线的潜在冲突。

**功耗优化方面**，深度睡眠模式的实现需要特别注意硬件资源的释放顺序。ISP 硬件句柄是文件作用域的静态变量，必须在进入睡眠前显式释放，否则下次唤醒后初始化会失败。触摸唤醒的防抖处理也经过了多次迭代——初始版本没有等待触摸释放和防抖，导致系统频繁被误唤醒。

**不足与反思**，当前系统的支付环节为模拟流程（自动确认），实际商用需要对接真实支付接口。语音识别在嘈杂环境下的准确率有待提升，后续可考虑增加唤醒词机制减少误触发。此外，商品出货的可靠性依赖于舵机的机械精度，实际部署时需要增加光电传感器检测商品是否成功掉落。

总体而言，本作品完整地覆盖了从感知到交互到执行的全链路，验证了 ESP32-P4 在边缘 AI 应用中的强大潜力。
